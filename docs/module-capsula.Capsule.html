Class: Capsule

(abstract) capsula.Capsule()

Capsule class is an abstract base class in the hierarchy of (built-in and user-defined) capsule classes. In many ways capsule classes are similar to the typical OO classes. They support single inheritance model and polymorphism the way OO classes do. However, capsules differ from OO classes in many ways as well. They are dynamic the way JavaScript language is. Also, they enforce a special encapsulation model and accommodate several new concepts which are explained in the lines that follow.

Capsule Properties

There are five types of capsule properties: parts, operations, methods, hooks, and loops.

Part is a property of capsule instance that represents another capsule instance that "belongs" to it, i.e. part capsule is basically a child capsule. Child capsule is always created in the context of its parent and accessible only from within that context. Next section explains the term capsule context more precisely.

An operation is a capsule's property that acts like a very powerful method. It can be used as a regular method, however operation adds support for asynchronous calls, argumetns filtering, declarative wiring to other operations to specify propagation of calls, etc. There are input and output operations. Read more on what can be achieved using operations.

Method is a capsule property that behaves as anyone would imagine, that is, as a simple JavaScript method. Methods can be either public or protected.

Hooks and loops provide capsule with a special support for managing hierarchy of external elements (widgets for example) in a flexible way.

When creating a capsule class, one must define which and how many of these properties should be present in it (see defCapsule). Once created, capsule class cannot be modified afterwards. All instances of that class would then have the properties as defined in the class. However, once an instance of capsule class is created, it can be dynamically modified (by adding new properties for example, or removing existing ones); just as anyone would expect from a JavaScript object.

Behind its interface made up of operations, public methods, hooks, and loops, capsules hide:

  • parts
  • the way parts' and capsule's operations are connected (wired) between one another,
  • the way parts' and capsule's hooks and loops are connected (tied) between one another,
  • protected methods (i.e. protected behavior), and
  • protected data.

Private properties are not supported.

Encapsulation Model and the "Current Context of Execution"

As a reminder, in a typical OO encapsulation model, all public properties and methods of all living objects are accessible from anywhere in the code, only a reference to an object is needed. Since everything public is accessible, we have to carefully manage references. Effectively managing references can be a bit too difficult and that is why capsules introduce quite a novel encapsulation model which is a restriction to the "everything public is accessible" policy.

To understand the policy of our encapsulation model, it is fundamentally important to understand the notion of "current context of execution". Current context of execution is specified with respect to the given point in time and it always represents a capsule instance:

For the given point in time t and the given capsule instance x, x is the current context of execution at t, if and only if the call stack at t lists a method of x and no method of capsule instance other than x deeper in the call stack.

If x is the current context of execution, we informally say that "we are in the context of x".

Now we can define "is part of" relation between capsules instances:

Two capsule instances x and y are in "is part of" relation (x is part of y) if and only if y was the current context of execution at the moment of instantiation of x.

Note that "is part of" relation is neither symmetric, nor transitive, nor reflexive.

For these two capsule instances we informally say that x is part of y. If x is part of y, we also informally say that "y is the owner of x". At any point in time a capsule instance can have zero or more parts (part properties) and zero or one owner.

Now, our policy could easily be expressed: when we are in the context of y, it is allowed to use (access, call) a) public properties of capsule instance x if and only if x is a part of y and b) protected properties of capsule instance y.

When trying to use a properties against these rules, an OUT_OF_CONTEXT error is going to be thrown.

How to create a capsule class?

To create a capsule class call the defCapsule method: 

var MyCapsuleClass = capsula.defCapsule(capsule_definition_object);

How to create a capsule instance?

To instantiate capsule class (i.e. to obtain a capsule instance or capsule) use one of the two possible ways:

a) instantiate imperatively using the new operator and a function returned from the call to defCapsule: 

var myCapsuleInstance = new MyCapsuleClass(optional_arguments);
In this case, myCapsuleInstance capsule implicitly becomes a part property of the capsule which represents the current context of execution.

b) instantiate declaratively using the capsule definition object in defCapsule: 

var OwnerClass = capsula.defCapsule({
    ...
    myCapsuleInstance: {
        capsule: MyCapsuleClass,
        args: [optional_arguments]
    }
    ...
});
In this case, we declare that every instance of the OwnerClass would have an instance of MyCapsuleClass (myCapsuleInstance) as its part property.

detach()

Detaches this capsule as a part of the capsule that represents the current context of execution. Destroys an implicit reference between the two capsules. Detaching the capsule also means destroying all the wires and ties that exist between that capsule and all other sibling capsules (including itself) as well as between that capsule and its owner capsule.

Recursively calls onDetach method (or operation) on this capsule and all its descendent capsules if they have that method (this is to release resources and prevent memory leak).

Example

Example of using superior

    base: BaseCapsule,
    '+ myMethod': function(){
        this.superior().myMethod.call(...); // calling a method that is just being overridden
    }
});
